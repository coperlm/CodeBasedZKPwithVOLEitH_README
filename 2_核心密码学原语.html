模块二：核心密码学原语——AFS 哈希与 McEliece 加密

一、概述

本模块提供两种基于编码的密码学原语，它们是整个 ZK 签名系统上层应用的代数骨架：

  1. AFS 哈希（AFC Hash）——一种基于 2-RNSD (2-Regular Null Syndrome Decoding) 困难问题的抗碰撞哈希函数。它的输出是一个二元向量，任何碰撞都等价于求解一个被公认为 NP-hard 的编码论问题。
  2. McEliece 加密——基于 不可区分置换 和 Goppa 码 的公钥加密方案（这里用于群签名中的身份密文生成），其安全性可归约到经典 McEliece 难题。

二、AFS 哈希函数：afs_hash.rs

2.1 数学定义

AFS 哈希函数定义为：

$$ h_B(u_0, u_1) = B_0 \cdot \text{RE}(u_0) \oplus B_1 \cdot \text{RE}(u_1) $$

其中：

  • $B_0, B_1 \in \mathbb{F}_2^{n \times m}$ 是两个伪随机二元矩阵
  • $u_0, u_1 \in \mathbb{F}_2^{n}$ 是 $n$ 比特的输入
  • $\text{RE}: \mathbb{F}_2^{n} \to \mathbb{F}_2^{m}$ 是正则编码函数（Regular Encoding），将 $n$ 比特输入映射为 $m = \frac{n}{c} \cdot 2^c$ 比特的稀疏二元向量，其汉明重量恰为 $n/c$

抗碰撞安全性：寻找 $h_B(u_0, u_1) = h_B(u’_0, u’_1)$ 的碰撞等价于在 $\mathbb{F}_2$ 上求解带噪的线性方程组（即 2-RNSD 问题），这是码基密码学中公认的困难问题。

2.2 正则编码 (Regular Encoding) 详解

2.2.1 数学定义

令 $c$ 为块大小（chunk size），$n$ 必须能被 $c$ 整除。正则编码将输入 $x$ 划分为 $\frac{n}{c}$ 个各长为 $c$ 比特的块。每个块 $x^{(i)} \in \mathbb{F}_2^{c}$ 被映射为一个独热向量（one-hot vector）：

$$ \text{RE}(x)i = e{\text{idx}(x^{(i)})} \in \mathbb{F}_2^{2^c} $$

其中 $\text{idx}(x^{(i)})$ 是 $c$ 比特的大端（big-endian）整数解释，$e_j$ 是仅在位置 $j$ 处为 1 的单位向量。整体输出由所有独热向量拼接而成：

$$ \text{RE}(x) = \left[e_{\text{idx}(x^{(0)})} ;|; e_{\text{idx}(x^{(1)})} ;|; \cdots ;|; e_{\text{idx}(x^{(n/c-1)})}\right] \in \mathbb{F}_2^{\frac{n}{c} \cdot 2^c} $$

2.2.2 公式-代码映射

数学符号代码中对应说明
$x \in \mathbb{F}_2^{n}$x: &MainBitSlice输入比特切片
$c$（块大小）c: usize每个块包含的比特数
域大小 $2^c$domain_size = 1 << c每个独热编码的维度

2.2.3 bits_to_index_be 的数学语义

#![allow(unused)]
fn main() {
pub(crate) fn bits_to_index_be(bits: &MainBitSlice) -> usize {
    bits.iter()
        .fold(0usize, |acc, bit| (acc << 1) | usize::from(*bit))
}
}

这实现了：

$$ \text{idx}(x^{(i)}) = \sum_{t=0}^{c-1} x^{(i)}_t \cdot 2^{c-1-t} $$

其中 $x^{(i)}_t$ 是第 $i$ 个块的第 $t$ 个比特。采用大端顺序意味着第 0 个比特是最高有效位。

2.2.4 常数时间版本 regular_encode_ct

常规的 regular_encode 使用条件分支（encoded.set(...)），其内存访问模式依赖于秘密的块值，存在侧信道泄漏风险。常数时间版本 regular_encode_ct 采用以下对策：

for 每个块 i:
    one_hot_idx ← bits_to_index_be(块)
    for 每个表索引 table_idx ∈ [0, domain_size):
        mask ← ct_eq_usize(table_idx, one_hot_idx)  // 常数时间相等测试
        encoded_bytes[pos] |= mask << bit_offset

关键点：

  • 扫描了所有 domain_size 个位置，无论结果如何
  • ct_eq_usize 使用纯位运算（无分支）实现安全相等判断：
#![allow(unused)]
fn main() {
fn ct_eq_usize(lhs: usize, rhs: usize) -> Choice {
    let diff = lhs ^ rhs;
    let bit = ((((diff | diff.wrapping_neg()) >> (usize::BITS - 1)) as u8) ^ 1) & 1;
    Choice::from(bit)
}
}

原理：$a = b \iff a \oplus b = 0 \iff (a \oplus b) \mid -(a \oplus b)$ 的最高位为 0。

2.3 AFS 哈希核心算法

2.3.1 朴素实现：afs_hash

#![allow(unused)]
fn main() {
pub fn afs_hash(b_cols: &[MainBitVec], x: &MainBitSlice, c: usize) -> Result<MainBitVec, String>
}

算法伪代码：

Algorithm: afs_hash(B, x, c)
Input:  B ∈ (F_2^n)^{m'}  — 列主序存储 (m' = (n/c)·2^c)
        x ∈ F_2^{n}       — 输入
        c ∈ ℕ              — 块大小
Output: y ∈ F_2^{rows(B)} — 哈希输出

1.  acc ← 0^n                           // 初始化为全零
2.  for each chunk x^{(i)} in x:        // 遍历每个 c 比特块
3.      col_idx ← i · 2^c + idx(x^{(i)}) // 计算全局列索引
4.      acc ← acc ⊕ B[:, col_idx]        // XOR 累加对应的列
5.  return acc

数学等价性：

$$ y = B \cdot \text{RE}(x) = \bigoplus_{i=0}^{n/c-1} B_{:, i \cdot 2^c + \text{idx}(x^{(i)})} $$

由于 $\text{RE}(x)$ 的每个块中仅有一个比特为 1，矩阵-向量乘法退化到只需选取 $n/c$ 个列做 XOR——这是稀疏矩阵的计算优势的又一次体现。

2.3.2 流式版本：afs_hash_streaming

#![allow(unused)]
fn main() {
pub fn afs_hash_streaming(
    b_cols: &StreamingMatrixCols,
    x: &MainBitSlice,
    c: usize,
) -> Result<MainBitVec, String>
}

与 afs_hash 的区别：矩阵 $B$ 不预先物化，而是通过 StreamingMatrixCols 按需流式生成每一列。在累加循环内调用 fill_column_bytes_prechecked，生成的列数据写入临时缓冲区、与累加器 XOR 后即丢弃。内存占用从物化矩阵的 $\mathcal{O}(nm)$ 降至临时缓冲区的 $\mathcal{O}(n)$。

2.3.3 常数时间流式版本：afs_hash_streaming_ct

安全问题：afs_hash_streaming 的列选择直接依赖于秘密输入 $x$ 的块值——这通过控制流暴露了索引信息。

常数时间对策：扫描每个块对应的所有 $2^c$ 个可能的列，但对每一列使用一个掩码来决定是否 XOR：

Algorithm: afs_hash_streaming_ct(B, x, c)
Input:  B ∈ StreamingMatrixCols
        x ∈ F_2^{n}
        c ∈ ℕ
Output: y ∈ F_2^{rows(B)}

1.  acc ← 0^byte_len
2.  for each chunk i in x:
3.      secret_idx ← idx(x^{(i)})
4.      for each table_idx ∈ [0, 2^c):
5.          column ← B[:, i·2^c + table_idx]   // 无条件读取所有列
6.          mask ← 0 - ct_eq(table_idx, secret_idx)  // 全0/全1掩码
7.          acc ← acc ⊕ (column & mask)         // 带掩码 XOR
8.  return acc

这里 masked_xor_assign_bytes 执行：

$$ \text{acc}[j] \leftarrow \text{acc}[j] \oplus (\text{column}[j] \land \text{mask}) $$

其中 mask 在 table_idx == secret_idx 时为 0xFF，否则为 0x00。因此内存访问模式（读取了所有列）和数据流（通过掩码进行条件 XOR）均不依赖秘密值。

并行加速：当矩阵规模超过阈值（CT_AFS_PAR_COL_THRESHOLD = 4096 列）且 Rayon 可用时，任务被分割为 chunk_count / 4 个并行子任务，每个子任务独立计算部分累加结果，最后通过 reduce 做 XOR 归并：

并行:  acc = acc_0 ⊕ acc_1 ⊕ ... ⊕ acc_{T-1}

2.4 普通线性矩阵-向量乘法

afs_hash 是针对正则编码模式（每块恰好选一列）的优化。对于一般的 $\mathbb{F}_2$ 线性变换：

$$ y = M \cdot v, \quad M \in \mathbb{F}_2^{n \times m}, v \in \mathbb{F}_2^{m} $$

代码提供了 linear_matrix_vector_mul_streaming 和其常数时间版本 linear_matrix_vector_mul_streaming_ct。两者的区别与 AFS 哈希中的对应版本相同：前者利用 $v$ 的稀疏性（只迭代为 1 的位），后者扫描所有列并依赖位掩码。

三、McEliece 加密与随机线性码：mceliece.rs

3.1 数学背景

Classic McEliece 使用 不可约 Goppa 码。公钥是一个伪随机的 $k \times (n-k)$ 二元生成矩阵 $G$。加密公式为：

$$ c = G \cdot \binom{u}{m} \oplus \text{RE}(e’) $$

其中：

  • $G \in \mathbb{F}_2^{(n-k) \times k}$ 是 McEliece 生成矩阵（公钥）
  • $u \in \mathbb{F}_2^{k - \ell}$ 是随机数
  • $m \in \mathbb{F}_2^{\ell}$ 是明文（对于群签名场景，$\ell$ 是用户索引的比特长度）
  • $e’ \in \mathbb{F}_2^{k’}$ 是噪声种子
  • $\text{RE}(e’)$ 是正则编码后的噪声向量（汉明重量极低）

3.2 参数配置

#![allow(unused)]
fn main() {
pub const SYS_T: usize = 96;        // Goppa 码纠错能力
pub const GFBITS: usize = 13;       // GF(2^13) 的扩展度
pub const SYS_N: usize = 4608;      // 码长 n
pub const PK_NROWS: usize = 1248;   // 公钥行数 = n - k = t·m
pub const PK_NCOLS: usize = 3360;   // 公钥列数 = k
pub const PK_ROW_BYTES: usize = 420;// 每行的字节数
pub const PK_BYTES: usize = 524160; // 公钥总字节数 (~512 KB)
}

这些参数对应 Classic McEliece-460896f（NIST 第三轮候选参数集），提供 128-bit 安全强度。

注：虽然 mceliece.rs 封装了 NIST 标准参数以供提取公钥结构，但在本系统实际的群签名应用中，为了满足正则噪声输出维度与密文维度严格对齐的要求，实际实例化的参数调整为 $n_e = 4096, k_e = 3328, t_e = 64$。

3.3 公式-代码映射

3.3.1 生成矩阵提取

在 run_matrix_validation 测试函数中，公钥的字节流被解析为 $G’$ 矩阵：

数学符号代码中对应
公钥字节流 $\text{pk} \in \mathbb{F}_2^{(n-k) \times k}$pk_bytes: &[u8]（长度 524160）
$G’$ 矩阵第 $i$ 行g_prime[i] = BitVec（截取前 PK_NCOLS 位）
$G’$ 的密度density ≈ 0.5（验证伪随机性质）

提取过程：

for row in 0..PK_NROWS:
    从 pk_bytes 读取连续 PK_ROW_BYTES 字节
    转为 BitVec
    截断为 PK_NCOLS 位  →  得到 G'[row]

3.3.2 矩阵-向量乘法（参考实现）

#![allow(unused)]
fn main() {
let mut y = bitvec![u8, Msb0; 0; PK_NCOLS];
for (row_idx, x_bit) in x.iter().enumerate() {
    if *x_bit {
        for col in 0..PK_NCOLS {
            if g_prime[row_idx][col] {
                let current = y[col];
                y.set(col, !current);
            }
        }
    }
}
}

这实现 $y = G’^{\top} \cdot x$（转置乘法），即：

  • $x$ 的每个为 1 的位置对应 $G’$ 的一行
  • 该行所有为 1 的列对输出做 XOR 翻转

3.3.3 在群签名中的完整加密应用

群签名约束生成函数 build_group_signature_constraints 中将 McEliece 加密拆解为两个部分：

第一部分 —— 线性组合：$G \cdot \binom{r}{\text{bin}(j)}$

#![allow(unused)]
fn main() {
fn append_linear_matrix_row_terms(
    equation: &mut BTreeSet<Monomial>,
    g_cols: &[MainBitVec],       // 列主序的 G 矩阵
    col_range: Range<usize>,     // 涉及的列范围
    var_start: usize,            // 证明变量起始索引
    row_idx: usize,              // 输出的第 row_idx 位
) {
    for (var_offset, col_idx) in col_range.enumerate() {
        if g_cols[col_idx][row_idx] {         // 如果 G 的该列在该行为 1
            xor_insert(equation, vec![var_start + var_offset]);  // 添加线性单项
        }
    }
}
}

这为密文 $c$ 的第 $i$ 位构建线性约束：

$$ c_i = \bigoplus_{j: G_{i,j} = 1} x_j $$

其中 $x = (r ;|; \text{bin}(j))$ 是随机数和用户索引的拼接。

第二部分 —— 正则噪声项：$\text{RE}(e’)$

    local_monomials: &[Vec<usize>],

}
}

这为密文的第 $i$ 位添加 $\text{RE}(e’)$ 贡献的噪声多项式。

完整密文约束：

$$ c_{1,i} = \left(\bigoplus_{j: G_{1,i,j}=1} (r_1 ;|; \text{bin}(j))_j\right) \oplus \text{RE}(e’_1)_i $$

$$ c_{2,i} = \left(\bigoplus_{j: G_{2,i,j}=1} (r_2 ;|; \text{bin}(\text{msg}))_j\right) \oplus \text{RE}(e’_2)_i $$

这精确实现了 $c = G \cdot \binom{u}{m} \oplus \text{RE}(e’)$，其中 $G_1, G_2$ 是两个独立的 McEliece 生成矩阵。

四、AFS 哈希在 Merkle 树中的应用（环签名上下文）

为阐明这两个密码学原语在系统中的协作关系，以下简述其在上层协议中的调用方式：

环签名的 Merkle 树每一层使用 AFS 双矩阵哈希：

$$ v_\ell = h_B(v_{\ell+1}, w_{\ell+1}) = B_0 \cdot \text{RE}(v_{\ell+1}) \oplus B_1 \cdot \text{RE}(w_{\ell+1}) $$

  • $(v_{\ell+1}, w_{\ell+1})$ 分别是左右子节点
  • $B_0$ 编码左子节点的影响，$B_1$ 编码右子节点的影响
  • 最终根哈希 $u_0 = v_0$ 被公开作为哈希链承诺

而 McEliece 加密则在群签名中提供追踪机制：管理方可利用私钥解密密文 $(c_1, c_2)$ 获取签名者的身份索引 $j$ 与消息 $m$，实现匿名性中的可控可追踪。
