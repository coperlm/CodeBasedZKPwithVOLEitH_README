## 声明

1. 当前签名使用的参数与论文中的 128-bit 配置基本对应；但由 `security.rs` 计算的关系证明上界不是严格 128 bits，具体数值见模块八；
2. 本仓库的实际签名体积和论文存在出入，但在实际的可接受范围内，详见性能测试部分；
3. 本仓库实现了在线签名和证明验证路径，但未经过生产级密码学审计，数据只用于算法正确性和效率的研究性验证；
4. 当前可执行路径是 128-bit profile；256-bit 参数只用于公式和表格复现，没有对应的可执行路径；
5. 单元测试、负向测试和跨进程测试可以支持实现级正确性判断，但不能替代独立密码学审计或安全证明。

## 2026-07-16 可复现端到端结果

下面结果由当前 `scripts/paper_speed_compare.sh` 生成，而非由上面的历史均值反推。每个 case 独立运行 10 次，去掉 1 个最高值和 1 个最低值，报告中间 8 次均值。计时范围为：

- Sign：FAEST `vole_commit`、BAVC opening、关系响应、grinding 与 canonical encoding；
- Verify：canonical decoding、FAEST `vole_reconstruct`、Fiat-Shamir 检查与关系验证。

本次运行命令等价于：

```bash
PAPER_SPEED_REPEATS=10 \
PAPER_SPEED_SINGLE_CPUSET=0 \
PAPER_SPEED_ALL_CPUSET=0-15 \
PAPER_SPEED_COMPARE_OUT_DIR=target/paper_speed_comparison_2026-07-16 \
bash scripts/paper_speed_compare.sh
```

### 与历史单 worker 汇总的差异

在相同方案、参数和 wire size 下，新单 worker 结果系统性快于历史单 worker 汇总：Sign 低 50.3%--59.7%（ReSolveD+ 为 50.3%），Verify 低 33.1%--73.1%。所有对应方案的 wire size 与历史表一致，说明签名格式和参数规模没有变化。

这不是可直接发表的“性能提升”结论：历史表没有保存其精确源码 revision、线程绑定、原始样本、完整编译选项和环境 metadata。新记录使用 commit `8dbd1c8`、`target-cpu=native`、CPU 0 affinity 和逐次样本；历史记录只标注了 `x86-64-v3` 等摘要信息。因此可以确认两组数据不应混用，但不能严谨地把全部差异归因于某一项代码优化。

## 数据的直接观察

1. `ell` 从 5 增加到 10 时，RingSig、Scope-LRS、GroupSig 和 FDABS 的 witness 长度 `l`、约束数 `t` 与签名体积都明显增加；这反映的是认证路径/电路关系规模的增加，而不是单纯的哈希调用次数变化。
2. 全部逻辑 CPU 的结果通常低于单核运行结果，但 Sign 和 Verify 的缩放并不相同：Sign 包含 VOLE commit、建树、关系响应和 opening，Verify 包含 reconstruct 和关系验证，受内存带宽及并行粒度的影响不同。
3. ReSolveD+ 的签名体积约为 3 KiB；其余应用的体积主要随 VOLE opening、witness correction、证明系数和应用关系数据增长。这里的体积是该次实验的签名表示大小，不应与论文中的纯理论通信量直接等同。
4. Scope-LRS 是仓库扩展，不能用它的速度或体积数据替代论文主方案的实验结果。

若重新生成结果，应记录 `RAYON_NUM_THREADS`、`RUST_TEST_THREADS`、CPU 指令集、源码 commit、FAEST revision、重复次数和原始样本；仅保留平均值不足以复现实验。本次文档修订时的源码提交为 `8dbd1c8`，FAEST 子模块为 `55b52681b4d962df9f55f4e403a68bca98b10d37`（v2.0.5），但这不能反向补足本表的历史 raw samples。
