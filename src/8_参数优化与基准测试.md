# 模块八：参数配置、工程优化与基准测试——安全分析与性能报告

## 一、概述

本模块是整个系统的工程化结晶，涵盖三个核心维度：

1. `security.rs` — 严格的安全性分析和参数选择，确保系统达到目标安全强度
2. `artifact_bundle.rs` — 极紧凑的签名序列化格式，将签名体积压缩到 KB 级别
3. `benches/paper_benchmarks.rs` + `scripts/` — 端到端基准测试框架和性能调优

## 二、安全参数分析：`security.rs`

### 2.1 安全性理论基础

$\Pi^t_{dD-Rep}$ 协议的安全性误差（soundness error）的上界为：

$$
\varepsilon \le 2^{-\text{combination\_bits}} + d \cdot 2^{-\Delta\_\text{bits}}
$$

其中：
- $\text{combination\_bits}$：挑战空间 $\chi_i$ 的比特数（即 $t = n$ 个挑战的域大小）
- $\Delta\_\text{bits}$：Verifier 的差值挑战 $\Delta$ 的比特数
- $d$：多项式的度数

### 2.2 安全性估计函数

```rust
pub fn polynomial_soundness_bound(
    degree: usize,
    combination_bits: usize,
    delta_bits: usize,
) -> Result<SoundnessEstimate, String> {
    let first_log2 = -(combination_bits as f64);               // log2(2^{-C})
    let second_log2 = (degree as f64).log2() - delta_bits as f64; // log2(d · 2^{-Δ})
    let max_log2 = first_log2.max(second_log2);
    let error_log2 = max_log2 + log2(2^{first-max} + 2^{second-max});
    Ok(SoundnessEstimate {
        degree, combination_bits, delta_bits,
        bit_security: -error_log2,
    })
}
```

使用数值稳定的 LogSumExp 技巧避免直接计算 $2^{-128} + d \cdot 2^{-128}$ 时的浮点下溢。

### 2.3 关键安全参数

`polynomial_soundness_bound` 确认了严格 128-bit 安全性无法达到——即使在理想情况下（两个域都是 128 比特），Forster 的上界也给出了约 124.68 比特的安全强度：

```
degree=9, combination=128, delta=128 → bit_security ≈ 124.678
degree=9, combination=128, delta=120 → bit_security ≈ 116.829
```

这与论文表 6 的 116-bit 保守估计一致。

Grinding 补偿：$w_\text{grind} = 8$ 比特的 grinding 提供了额外的 $\log_2(2^8) = 8$ 比特工作因子：

```rust
pub fn optimistic_work_factor_bits(self, grinding_bits: usize) -> f64 {
    self.bit_security + grinding_bits as f64
}
```

这使得实际攻击难度达到约 $116.8 + 8 \approx 124.8$ 比特。

### 2.4 参数套件

| 方案 | $n$ | $c$ | $d$ | 挑战域 | $\Delta$ 预估比特 | $w_\text{grind}$ | 安全估计算 |
|------|-----|-----|-----|--------|------------|:---:|-----------|
| ReSolveD+（论文表 6 Var2） | 453 | 3 | 3 | 128 | 120 | 8 | ~118.4 + 8 |
| 环签名 / 群签名 / FDABS | 1280 | 8 | 9 | 128 | 120 | 8 | ~116.8 + 8 |

环签名等方案度数 $d = c+1 = 9$（因方向比特多一度），但安全分析仍适用相同的 bound 公式。

## 三、极紧凑序列化：`artifact_bundle.rs`

### 3.1 签名体积的理论基础

传统基于 Stern 协议的 ZKP 签名：每次 repetition 需要约 $k$ 比特的承诺 + 打开数据，重复 $\lambda$ 次（$\lambda = 128$），总签名体积通常在 兆字节（MB）级别。

VOLEitH 方法：通过 $\tau$ 棵 GGM 树并行承诺 + 选择性打开，签名体积压缩到 KB 级别。核心压缩机制：

| 组件 | 包含内容 | 典型大小 |
|------|---------|---------|
| Commitment Hash | 32 字节 | 32 B |
| IV | 16 字节 | 16 B |
| Consistency C | $(\tau-1) \cdot \lceil \hat{\ell} / 8 \rceil$ | ~1.5 KB |
| Ũ | `FAEST_VOLE_HASH_OUTPUT_BYTES` = 18 字节 | 18 B |
| Opening | 叶子打开数据 | ~1.5 KB |
| Challenge | 16 字节 | 16 B |
| Grind Counter | 4 字节 | 4 B |
| VOLE subtotal | | ~3 KB |
| Correction | $k$ 比特 | ~56 B |
| Proof Coeffs | $d$ 个 $\mathbb{F}\_{2^{128}}$ 元素 | $d \times 16$ B |
| Signature total | | ~3159 B |

`artifact_bundle.rs` 中的常量：

```rust
pub const ARTIFACT_K: usize = 453;
pub const ARTIFACT_C: usize = 3;
pub const ARTIFACT_ROWS: usize = 453;
pub const ARTIFACT_SIGNATURE_BYTES: usize = 3159;  // 仅 3.1 KB!
```

### 3.2 压缩策略详解

#### 策略一：种子代替矩阵

公钥中的 $B$ 矩阵不序列化 $n \times m$ 比特（$453 \times 1208 \approx 547$ Kb $\approx 68$ KB），而是仅序列化 16 字节的种子 $\sigma$：

```rust
bytes.extend_from_slice(&public_key.b_cols.seed());  // 仅 16 字节
```

Verifier 在验证时用 `StreamingMatrixCols::new(seed, rows, cols)` 重新生成矩阵。

#### 策略二：紧凑位编码

所有比特向量使用精确比特数编码（`canonical_bit_bytes`），移除尾部未使用比特：

```rust
fn canonical_bit_bytes(bits: &MainBitVec) -> Vec<u8> {
    let mut bytes = bits.clone().into_vec();
    bytes.truncate(bits.len().div_ceil(8));
    if bits.len() % 8 != 0 && !bytes.is_empty() {
        bytes[last] &= 0xFFu8 << (8 - (bits.len() % 8));  // 清除尾部未使用比特
    }
    bytes
}
```

#### 策略三：直接合并 VOLE 哈希

Ũ 只有 18 字节——这是 FAEST VOLE 哈希 $H$ 在 $\mathbb{F}\_{2^{128}}$ 上的输出截断为 $128 + 16$ 比特：

```rust
pub(crate) const FAEST_VOLE_HASH_OUTPUT_BYTES: usize = 18; // 128 + 16 比特
```

#### 策略四：固定且极少的盲化系数

不同方案的 proof 系数数量：

| 方案 | 系数数量 | 总大小 |
|------|:--------:|:------:|
| ReSolveD+ ($c=3$) | 3 | 48 B |
| 环签名 ($c=8$) | 9 | 144 B |
| 群签名 ($c_\text{max}=8$) | 9 | 144 B |
| FDABS ($c=8$) | 9 | 144 B |

相比传统方法中每个约束都需要一个系数，这里仅需要 $d$ 个 $\mathbb{F}\_{2^{128}}$ 元素。

### 3.3 格式版本化和严格验证

```rust
const MAGIC: &[u8; 8] = b"CBZKRPB1";  // ReSolveD+ Bundle 魔术字
const VERSION: u16 = 1;
```

- 魔术字：8 字节 `CBZKRPB1` 用于快速识别格式
- 版本号：支持未来扩展
- 尾部拒绝：`decoder.finish()` 检查无多余字节，防止格式混淆攻击
- 参数硬编码检查：解码时验证 `(k, c, rows, cols, y_bits)` 必须匹配常量，防止跨参数集误用

### 4.3 `par_chunks_mut` 的列并行

在 χ' 计算的热路径上，使用 `par_chunks_mut` 将列划分为连续 chunk，每个 worker 拥有独立的 `column_bytes` 缓冲区：

```rust
fn compute_chi_prime_streaming_f128_range_with_table(..., col_start, col_count) {
    if col_count >= CHI_PRIME_PAR_COL_THRESHOLD {
        chi_prime.par_chunks_mut(CHI_PRIME_PAR_COL_CHUNK)
            .enumerate()
            .for_each_init(
                || vec![0u8; byte_len],  // 每个 worker 独立缓冲区
                |column_bytes, (chunk_idx, chunk)| {
                    for (offset, chi) in chunk.iter_mut().enumerate() {
                        b_cols.fill_column_bytes_prechecked(base_col + offset, column_bytes);
                        *chi = subset_xor_from_bytes(column_bytes, table, byte_len);
                    }
                },
            );
    }
}
```

关键设计：`for_each_init` 为每个 Rayon worker 分配独立的列字节缓冲区，避免锁竞争和缓存行乒乓。

### 4.5 微架构级优化

手工循环展开：`xor_32!` 宏和 `xor_pair_32!` 宏一次性处理 32 字节（160 字节总长时展开为 5 块），减少循环分支开销：

```rust
// 160 字节的特殊优化路径
if byte_len == 160 {
    xor_32!(acc, &tables[0..32], &bytes[0..32]);
    xor_32!(acc, &tables[32..64], &bytes[32..64]);
    // ...
    xor_32!(acc, &tables[128..160], &bytes[128..160]);
}
```

栈分配工作区：对于 $2^c \le 256$ 的小域，蝴蝶求值使用栈分配的 `[F; 256]` 定长数组，避免热路径上的堆分配：

```rust
if domain_size <= STACK_BUTTERFLY_DOMAIN {  // 256
    let mut work: [F; STACK_BUTTERFLY_DOMAIN] = std::array::from_fn(|_| F::ZERO);
    return evaluate_chunk_butterfly_reuse_slice(..., &mut work[..domain_size]);
}
```

四累加器线性和：`linear_weighted_sum` 使用 4 路累加器减少 $\mathbb{F}\_{2^{128}}$ 乘法的数据依赖链长度：

```rust
let mut acc0 = F128b::ZERO;
let mut acc1 = F128b::ZERO;
let mut acc2 = F128b::ZERO;
let mut acc3 = F128b::ZERO;
while idx < bulk_len {
    acc0 += coeffs[idx] * affine[idx];
    acc1 += coeffs[idx+1] * affine[idx+1];
    acc2 += coeffs[idx+2] * affine[idx+2];
    acc3 += coeffs[idx+3] * affine[idx+3];
    idx += 4;
}
let total = (acc0 + acc1) + (acc2 + acc3);
```

## 五、工程优化总结

| 优化层次 | 具体技术 | 影响 |
|---------|---------|------|
| 算法 | 隐式求值/蝴蝶折叠 | $O(t \cdot 2^c) \to O(m \cdot c)$ |
| 内存 | `StreamingMatrixCols` 流式列生成 | 矩阵内存 $O(nm) \to O(1)$ |
| 序列化 | 种子代替矩阵、紧凑位编码 | 签名体积 MB → KB |
| 并行 | Rayon `par_iter` / `par_chunks_mut` / `join` | 接近核数线性加速 |
| 硬件 | PCLMULQDQ / AES-NI / BMI2 | 亚微秒级域运算 |
| 微架构 | 循环展开、栈分配、四累加器 | 额外 2-3$\times$ 系数提升 |
| 安全 | Grinding 8-bit、常数时间实现 | 124-bit 等效安全强度 |
| 工程 | 双后端切换（HW/SW fallback） | 跨平台兼容 |
