Code-Based Zero-Knowledge from VOLE-in-the-Head: 核心代码与架构解析报告

📑 报告导读

模块一：底层数学与内存优化——GF(2¹²⁸) 域运算与大矩阵流式计算

一、概述

本模块是整个 ZKP 系统的数学基石与性能瓶颈突破口。它解决两个根本问题：

  1. 如何在 $\mathbb{F}_{2^{128}}$ 上做极速乘法——零知识证明协议中大量涉及域上的线性组合与多项式求值，域乘法的速度直接决定协议延迟。
  2. 如何在不耗尽内存的前提下操作 $\mathbb{F}_2^{n \times m}$（$n \approx 10^4, m \approx 10^5$）量级的二元矩阵——基于编码的 ZK 协议需要大量的随机矩阵（如 $B \in \mathbb{F}_2^{n \times m}$），直接物化（materialize）会导致数十 GB 的内存占用。

二、GF(2¹²⁸) 有限域运算：finite_field.rs

2.1 数学背景

$\text{GF}(2^{128})$ 是特征为 2 的扩域，域元素是次数小于 128 的二元多项式：

$$ a(x) = a_{127}x^{127} + a_{126}x^{126} + \cdots + a_1x + a_0, \quad a_i \in \mathbb{F}_2 $$

加法即按位 XOR；乘法是多项式乘法再模一个 128 次不可约多项式（通常是 $x^{128} + x^7 + x^2 + x + 1$，即 GCM 所用的 reduction 多项式）。

2.2 代码实现方式

核心实现路径：

代码没有自己实现域运算，而是复用 Galois Inc. 的 swanky 密码库（原 scuttlebutt 库）。swanky-field-binary 在编译时通过条件编译自动检测目标架构：

  • x86_64：使用 PCLMULQDQ 指令 + AES-NI 的 AESDEC 做多项式约简——单条指令在一个时钟周期内完成 64-bit 无进位乘法，约简阶段复用 AES 轮函数加速。
  • aarch64/ARM：使用 PMULL 指令做多项式乘法，配合 ARM 的 EOR3 等指令做约简。

从 benches/gf128_bench.rs 可以看到，当使用 RUSTFLAGS="-C target-cpu=native" 编译时，微基准测试报告显示平均每次乘法耗时在亚纳秒级（实测 < 0.5 ms / 万次），验证了硬件加速路径被正确激活。

2.3 安全注意事项

整个 finite_field.rs 仅 47 行有效代码（其余为测试），它不做任何内存分配或可变状态操作，因此侧信道攻击面极低。域乘法的常数时间性质由硬件指令（PCLMULQDQ / PMULL）天然保证——这些指令是固定延迟的，不依赖操作数的值。

三、大矩阵流式计算：streaming_matrix.rs

3.1 问题与设计思想

在基于编码的 ZK 协议中（AFC 哈希、VOLE 承诺等），需要大量使用伪随机矩阵：

$$ B \in \mathbb{F}_2^{n \times m}, \quad n = 1280,\ m = 81920 $$

若直接物化，$1280 \times 81920 \text{ bits} = 1280 \times 10240 \text{ bytes} \approx 12.5 \text{ MB}$。对于单张矩阵尚且可控，但协议运行中会生成几十到上百张这样的矩阵，加上中间变量的暂存，内存压力急剧上升。更关键的是，VOLE 树的叶子承诺需要用到种子衍生（seed-derived）的矩阵列——大量矩阵只在求值时用一次，无需常驻内存。

设计思路是 “列式伪随机流式生成”：不存储整个矩阵，而是存储一个 128-bit 种子，在需要某一列时即时计算。

3.4 内存管理策略详解

策略一：零物化（Zero Materialization）

矩阵从未完整存储在内存中。StreamingMatrixCols 仅持有 32 字节的状态信息（16 字节种子 + 部分元数据）。每次调用 column(j) 生成一列的 n 个比特（约 $n/8$ 字节），用完即丢弃。这使得 $1280 \times 81920$ 的矩阵实际占用从 $\approx 12.5\text{ MB}$ 降至 $\approx 0.00003\text{ MB}$。

策略二：延迟生成（Lazy Generation）

fill_column_bytes 使用按需填充模式。只有 column(j) 或 fill_column_bytes 被显式调用时，数据才被生成。materialize() 方法虽然提供了全矩阵物化的接口（用于测试），但在生产代码中不使用——生产中使用 iter() 逐个消费列。

策略三：AES-CTR 作为向量 PRG

每一列 $B_{:,j}$ 通过 AES-128 在 CTR 模式下生成：

$$ B_{:,j} = \text{AES}\sigma(\text{ctr}=j || \text{nonce}=0) ; | ; \text{AES}\sigma(\text{ctr}=j || \text{nonce}=1) ; | ; \cdots $$

其中 $\sigma$ 是 16 字节种子。每个 AES 块产生 128 位伪随机输出，因此生成 $n$ 位列向量需要 $\lceil n/128 \rceil$ 次 AES 加密调用。

关键优化细节：

  • 密钥扩展一次：expand_key 在构造 MatrixPrg 时把种子扩展为 11 轮子密钥，后续 fill_column_bytes 的每次 AES 加密都复用这些轮密钥（round_keys），避免重复扩展。
  • 向量化 AES：aes128_ctr::fill 在 while 循环中批量处理连续块的加密，利用 _mm_aesenc_si128 / _mm_aesenclast_si128 指令。在循环内部，每次迭代写入 16 字节到输出缓冲区，并递增 128-bit 计数器。剩余不满一块的尾部数据通过 copy_nonoverlapping 截取。
  • 尾部掩码清除：因 $n$ 可能不是 8 的倍数，最后一字节的高位可能包含无效比特。clear_column_tail_bits 用 tail_mask 清除这些位，保证列向量的有效性。

策略四：双后端切换

四、稀疏矩阵-向量乘法：bitvec_ops.rs

4.1 问题

在基于编码的 ZK 中，$y = B \cdot x$ 中 $x$ 是一个正则编码词（Regular Encoded word），其汉明重量极低——即 $x$ 中只有 $c$ 个比特为 1，其余全为 0。这意味着全矩阵乘法可以退化为稀疏乘法：

$$ y_i = \bigoplus_{j: x_j = 1} B_{i,j} $$

4.2 公式-代码映射

数学概念代码中对应
$B_{n \times m}$matrix_b: &[BitVec<u8, Msb0>] — 行优先 MSB 存储
$x$ 中为 1 的索引集合 ${j \mid x_j = 1}$x_indices: &[usize]
$y_i = \bigoplus_{j: x_j=1} B_{i,j}$对每个 col_idx，遍历所有行，异或对应比特
$\mathbb{F}_2$ 加法（XOR）result.set(row, !current)

4.3 算法伪代码

Algorithm: Matrix-Vector Multiply (Sparse)
Input:  matrix_b (rows × cols, row-major BitVec)
        x_indices (sorted list of indices where x_j = 1)
Output: result (len = rows, BitVec)

for each col_idx in x_indices:
    byte_pos  ← col_idx / 8           // 列所在字节位置
    bit_in_byte ← 7 - (col_idx % 8)   // Msb0 下比特在字节中的偏移
    mask ← 1 << bit_in_byte
    
    for row in 0..rows:
        if matrix_b[row].bytes[byte_pos] & mask != 0:
            result[row] ← NOT result[row]   // 即 XOR 1

4.4 缓存优化分析

该算法利用了矩阵的行主序存储（Vec<BitVec<u8, Msb0>>）与稀疏索引遍历的结合：

  1. 空间局部性：对于同一个 col_idx，连续访问 matrix_b[0..rows] 的同一字节位置 byte_pos，这些字节在内存中紧密排列（每行相邻），因此 CPU 缓存命中率极高。
  2. 稀疏性红利：若汉明重量 $w = 320$ 而列数 $m = 81920$，则避免了 $81920 - 320 \approx 81600$ 次全列扫描，加速比达到 $\mathcal{O}(m/w)$。
  3. 字节级寻址：通过预计算 byte_pos 和位掩码，避免了逐位索引的间接开销。

从测试代码 test_matrix_vector_multiply_report 的配置（1280 行 × 81920 列，汉明重量 320）看，平均一次乘法在数十毫秒量级（具体取决于 CPU 频率和缓存大小），这为上层协议的高效运行奠定了基础。

模块二：核心密码学原语——AFS 哈希与 McEliece 加密

一、概述

本模块提供两种基于编码的密码学原语，它们是整个 ZK 签名系统上层应用的代数骨架：

  1. AFS 哈希（AFC Hash）——一种基于 2-RNSD (2-Regular Null Syndrome Decoding) 困难问题的抗碰撞哈希函数。它的输出是一个二元向量，任何碰撞都等价于求解一个被公认为 NP-hard 的编码论问题。
  2. McEliece 加密——基于 不可区分置换 和 Goppa 码 的公钥加密方案（这里用于群签名中的身份密文生成），其安全性可归约到经典 McEliece 难题。

二、AFS 哈希函数：afs_hash.rs

2.1 数学定义

AFS 哈希函数定义为：

$$ h_B(u_0, u_1) = B_0 \cdot \text{RE}(u_0) \oplus B_1 \cdot \text{RE}(u_1) $$

其中：

  • $B_0, B_1 \in \mathbb{F}_2^{n \times m}$ 是两个伪随机二元矩阵
  • $u_0, u_1 \in \mathbb{F}_2^{n}$ 是 $n$ 比特的输入
  • $\text{RE}: \mathbb{F}_2^{n} \to \mathbb{F}_2^{m}$ 是正则编码函数（Regular Encoding），将 $n$ 比特输入映射为 $m = \frac{n}{c} \cdot 2^c$ 比特的稀疏二元向量，其汉明重量恰为 $n/c$

抗碰撞安全性：寻找 $h_B(u_0, u_1) = h_B(u’_0, u’_1)$ 的碰撞等价于在 $\mathbb{F}_2$ 上求解带噪的线性方程组（即 2-RNSD 问题），这是码基密码学中公认的困难问题。

2.2 正则编码 (Regular Encoding) 详解

2.2.1 数学定义

令 $c$ 为块大小（chunk size），$n$ 必须能被 $c$ 整除。正则编码将输入 $x$ 划分为 $\frac{n}{c}$ 个各长为 $c$ 比特的块。每个块 $x^{(i)} \in \mathbb{F}_2^{c}$ 被映射为一个独热向量（one-hot vector）：

$$ \text{RE}(x)i = e{\text{idx}(x^{(i)})} \in \mathbb{F}_2^{2^c} $$

其中 $\text{idx}(x^{(i)})$ 是 $c$ 比特的大端（big-endian）整数解释，$e_j$ 是仅在位置 $j$ 处为 1 的单位向量。整体输出由所有独热向量拼接而成：

$$ \text{RE}(x) = \left[e_{\text{idx}(x^{(0)})} ;|; e_{\text{idx}(x^{(1)})} ;|; \cdots ;|; e_{\text{idx}(x^{(n/c-1)})}\right] \in \mathbb{F}_2^{\frac{n}{c} \cdot 2^c} $$

2.2.3 bits_to_index_be 的数学语义

#![allow(unused)]
fn main() {
pub(crate) fn bits_to_index_be(bits: &MainBitSlice) -> usize {
    bits.iter()
        .fold(0usize, |acc, bit| (acc << 1) | usize::from(*bit))
}
}

这实现了：

$$ \text{idx}(x^{(i)}) = \sum_{t=0}^{c-1} x^{(i)}_t \cdot 2^{c-1-t} $$

其中 $x^{(i)}_t$ 是第 $i$ 个块的第 $t$ 个比特。采用大端顺序意味着第 0 个比特是最高有效位。

2.2.4 常数时间版本 regular_encode_ct

常规的 regular_encode 使用条件分支（encoded.set(...)），其内存访问模式依赖于秘密的块值，存在侧信道泄漏风险。常数时间版本 regular_encode_ct 采用以下对策：

for 每个块 i:
    one_hot_idx ← bits_to_index_be(块)
    for 每个表索引 table_idx ∈ [0, domain_size):
        mask ← ct_eq_usize(table_idx, one_hot_idx)  // 常数时间相等测试
        encoded_bytes[pos] |= mask << bit_offset

关键点：

  • 扫描了所有 domain_size 个位置，无论结果如何
  • ct_eq_usize 使用纯位运算（无分支）实现安全相等判断：
#![allow(unused)]
fn main() {
fn ct_eq_usize(lhs: usize, rhs: usize) -> Choice {
    let diff = lhs ^ rhs;
    let bit = ((((diff | diff.wrapping_neg()) >> (usize::BITS - 1)) as u8) ^ 1) & 1;
    Choice::from(bit)
}
}

原理：$a = b \iff a \oplus b = 0 \iff (a \oplus b) \mid -(a \oplus b)$ 的最高位为 0。

2.3 AFS 哈希核心算法

2.3.1 朴素实现：afs_hash

#![allow(unused)]
fn main() {
pub fn afs_hash(b_cols: &[MainBitVec], x: &MainBitSlice, c: usize) -> Result<MainBitVec, String>
}

算法伪代码：

Algorithm: afs_hash(B, x, c)
Input:  B ∈ (F_2^n)^{m'}  — 列主序存储 (m' = (n/c)·2^c)
        x ∈ F_2^{n}       — 输入
        c ∈ ℕ              — 块大小
Output: y ∈ F_2^{rows(B)} — 哈希输出

1.  acc ← 0^n                           // 初始化为全零
2.  for each chunk x^{(i)} in x:        // 遍历每个 c 比特块
3.      col_idx ← i · 2^c + idx(x^{(i)}) // 计算全局列索引
4.      acc ← acc ⊕ B[:, col_idx]        // XOR 累加对应的列
5.  return acc

数学等价性：

$$ y = B \cdot \text{RE}(x) = \bigoplus_{i=0}^{n/c-1} B_{:, i \cdot 2^c + \text{idx}(x^{(i)})} $$

由于 $\text{RE}(x)$ 的每个块中仅有一个比特为 1，矩阵-向量乘法退化到只需选取 $n/c$ 个列做 XOR——这是稀疏性红利的又一次体现。

2.3.2 流式版本：afs_hash_streaming

#![allow(unused)]
fn main() {
pub fn afs_hash_streaming(
    b_cols: &StreamingMatrixCols,
    x: &MainBitSlice,
    c: usize,
) -> Result<MainBitVec, String>
}

与 afs_hash 的区别：矩阵 $B$ 不预先物化，而是通过 StreamingMatrixCols 按需流式生成每一列。在累加循环内调用 fill_column_bytes_prechecked，生成的列数据写入临时缓冲区、与累加器 XOR 后即丢弃。内存占用从物化矩阵的 $\mathcal{O}(nm)$ 降至临时缓冲区的 $\mathcal{O}(n)$。

2.3.3 常数时间流式版本：afs_hash_streaming_ct

安全问题：afs_hash_streaming 的列选择直接依赖于秘密输入 $x$ 的块值——这通过控制流暴露了索引信息。

常数时间对策：扫描每个块对应的所有 $2^c$ 个可能的列，但对每一列使用一个掩码来决定是否 XOR：

Algorithm: afs_hash_streaming_ct(B, x, c)
Input:  B ∈ StreamingMatrixCols
        x ∈ F_2^{n}
        c ∈ ℕ
Output: y ∈ F_2^{rows(B)}

1.  acc ← 0^byte_len
2.  for each chunk i in x:
3.      secret_idx ← idx(x^{(i)})
4.      for each table_idx ∈ [0, 2^c):
5.          column ← B[:, i·2^c + table_idx]   // 无条件读取所有列
6.          mask ← 0 - ct_eq(table_idx, secret_idx)  // 全0/全1掩码
7.          acc ← acc ⊕ (column & mask)         // 带掩码 XOR
8.  return acc

这里 masked_xor_assign_bytes 执行：

$$ \text{acc}[j] \leftarrow \text{acc}[j] \oplus (\text{column}[j] \land \text{mask}) $$

其中 mask 在 table_idx == secret_idx 时为 0xFF，否则为 0x00。因此内存访问模式（读取了所有列）和数据流（通过掩码进行条件 XOR）均不依赖秘密值。

并行加速：当矩阵规模超过阈值（CT_AFS_PAR_COL_THRESHOLD = 4096 列）且 Rayon 可用时，任务被分割为 chunk_count / 4 个并行子任务，每个子任务独立计算部分累加结果，最后通过 reduce 做 XOR 归并：

并行:  acc = acc_0 ⊕ acc_1 ⊕ ... ⊕ acc_{T-1}

2.4 普通线性矩阵-向量乘法

afs_hash 是针对正则编码模式（每块恰好选一列）的优化。对于一般的 $\mathbb{F}_2$ 线性变换：

$$ y = M \cdot v, \quad M \in \mathbb{F}_2^{n \times m}, v \in \mathbb{F}_2^{m} $$

代码提供了 linear_matrix_vector_mul_streaming 和其常数时间版本 linear_matrix_vector_mul_streaming_ct。两者的区别与 AFS 哈希中的对应版本相同：前者利用 $v$ 的稀疏性（只迭代为 1 的位），后者扫描所有列并依赖位掩码。

三、McEliece 加密与随机线性码：mceliece.rs

3.1 数学背景

Classic McEliece 使用 不可约 Goppa 码。公钥是一个伪随机的 $k \times (n-k)$ 二元生成矩阵 $G$。加密公式为：

$$ c = G \cdot \binom{u}{m} \oplus \text{RE}(e’) $$

其中：

  • $G \in \mathbb{F}_2^{(n-k) \times k}$ 是 McEliece 生成矩阵（公钥）
  • $u \in \mathbb{F}_2^{k - \ell}$ 是随机数
  • $m \in \mathbb{F}_2^{\ell}$ 是明文（对于群签名场景，$\ell$ 是用户索引的比特长度）
  • $e’ \in \mathbb{F}_2^{k’}$ 是噪声种子
  • $\text{RE}(e’)$ 是正则编码后的噪声向量（汉明重量极低）

模块三：VOLEitH 状态机与跨语言 FFI

一、概述

本模块是整个 ZKP 系统的核心引擎——它负责将密码学约束系统与底层的延迟 VOLE 功能（Delayed VOLE functionality, $\mathcal{F}_{\text{sVOLE}}$）衔接起来。架构上采用两层级联设计：

  1. C 层（FAEST）：提供高效的 GGM 树形向量承诺（BAVC - Binary Auxiliary Vector Commitment），处理 $\tau$ 棵 $\lambda$ 深度的 GGM 树的构建、打开和验证。这是密码学计算密集部分。
  2. Rust 层：通过 FFI 安全封装 C 层状态机，并在此基础上实现 Prover/Verifier 的协议状态机，负责挑战生成、相关性提取（correlation extraction）、约束绑定等高层协议逻辑。

二、为什么需要 FFI 复用 FAEST 的 C 代码？

2.1 GGM 树形向量承诺（BAVC）的密码学角色

在 VOLE-in-the-Head 范式中，Prover 需要承诺 $\tau$ 个伪随机种子 $K_1, \dots, K_\tau$，然后对每个种子应用 GGM 伪随机函数树（Goldreich-Goldwasser-Micali 树）展开出 $\lambda$ 层的叶子。这些叶子形成了 VOLE 关系中 Prover 的秘密向量：

$$ {(u_i, v_i)}_{i=1}^\tau \xrightarrow{\text{GGM 展开}} {(u, \mathbf{v})} \in \mathbb{F}2^\ell \times \mathbb{F}{2^{128}}^\ell $$

BAVC（Binary Auxiliary Vector Commitment）是 FAEST 使用的一种优化的 GGM 树承诺方案，它支持：

  • 批量承诺：一次承诺 $\tau$ 棵树的所有根密钥
  • 选择性打开：根据 Verifier 挑战，仅打开挑战指定的叶子路径
  • 一致性验证：Verifier 可通过承诺哈希验证打开的叶子是否正确

2.2 不复用的代价

如果完全用 Rust 重新实现 BAVC：

  • 需要重新实现 $\tau$ 棵 GGM 树的构建、密钥扩展、Merkle 哈希聚合——这是数千行经过审计的 C 代码
  • 需要保证与 FAEST 规范的行为完全一致（包括常数时间执行、字节序、SHA-3 用法等）
  • FAEST 的 C 代码经历了 NIST 后量子密码标准化过程的严格审查，复用它直接继承了其安全保证

3.3.2 行指针数组的拼接

vole_commit 要求 v 参数为一个 *mut *mut u8（指向指针数组的指针），每个指针指向 VOLE 的 $\lambda$ 行中的一行。Rust 侧构建方式：

#![allow(unused)]
fn main() {
let mut v_rows = vec![0u8; public_params.lambda_bits * row_bytes];
let mut row_ptrs = (0..public_params.lambda_bits)
    .map(|row| unsafe { v_rows.as_mut_ptr().add(row * row_bytes) })
    .collect::<Vec<_>>();
}

C 函数写入 v_rows 后，Rust 侧通过 row_ptrs 重新解读为连续内存块中的 $\lambda$ 行 × row_bytes 列的矩阵。这样既满足了 C 接口的指针数组要求，又在 Rust 侧保持了连续内存的安全访问。

3.4 不依赖 C 的 Rust 原生实现：VOLE 哈希

为减少 C FFI 调用的性能开销和跨语言维护负担，FAEST 的 VOLE 哈希函数（vole_hash）在 Rust 中完整重新实现：

#![allow(unused)]
fn main() {
fn vole_hash_128f_rust(seed, input, hash_ell_bits) -> [u8; 18]
}

这实现了一个基于 F128b 域运算的密钥哈希（Keyed Hash），核心计算为：

$$ H = (h_2, h_3) = (r_0 \cdot h_0 + r_1 \cdot h_1,; r_2 \cdot h_0 + r_3 \cdot h_1) $$

其中：

  • $h_0$ 通过 Horner 法在 $\mathbb{F}_{2^{128}}$ 上计算多项式求值
  • $h_1$ 通过 Horner 法在 $\mathbb{F}_{2^{64}}$ 上计算多项式求值（使用 gf64_mul）

gf64_mul 的双重加速策略：

路径条件实现
硬件加速x86_64 + PCLMULQDQ_mm_clmulepi64_si128 一次完成 64×64→128 无进位乘法
可移植回退其他平台逐位 shift-and-XOR 软实现

硬件路径使用 PCLMULQDQ 指令完成 $\mathbb{F}_{2^{64}}$ 上的无进位多项式乘法，约简通过经典的两步折叠（fold）算法实现：

#![allow(unused)]
fn main() {
let overflow = (high >> 63) ^ (high >> 61) ^ (high >> 60);
low ^ high ^ (high << 1) ^ (high << 3) ^ (high << 4)
    ^ overflow ^ (overflow << 1) ^ (overflow << 3) ^ (overflow << 4)
}

这约简多项式 $x^{64} + x^4 + x^3 + x + 1$。

正确性保障：test_rust_vole_hash_matches_faest_c_reference 在多种输入长度（0 ~ 28176 比特）和多种随机种子下，逐字节比对 Rust 实现与 C 参考实现的输出，确保完全一致。

四、状态机设计：voleith.rs

4.1 延迟 VOLE 功能（Delayed VOLE）

在 VOLEitH 协议中，Prover 和 Verifier 需要建立以下 VOLE 关系：

$$ \text{Prover 持有: } (u, \mathbf{v}) \in \mathbb{F}2^\ell \times \mathbb{F}{2^{128}}^\ell $$ $$ \text{Verifier 持有: } (\Delta, \mathbf{q}) \in \mathbb{F}{2^{128}} \times \mathbb{F}{2^{128}}^\ell $$

满足线性关系：

$$ \mathbf{v} = \mathbf{q} + \Delta \cdot u $$

其中 $\Delta$ 是 Verifier 的全局挑战，$u$ 是 Prover 的秘密比特向量，$\mathbf{v}, \mathbf{q}$ 是对应的 $\mathbb{F}_{2^{128}}$ 域元素向量。

“延迟“的含义是：在协议初始阶段（承诺阶段），Prover 和 Verifier 仅交换承诺，等到协议运行到特定步骤后才确定 $\Delta$ 和挑战。

       ell_hat_bits, commitment_hash, iv, consistency_c)
squeeze() → challenge_1

第 2 轮关系挑战 relation_seed：源自承诺哈希 + VOLE 哈希值 + 校正子

absorb(profile, statement, witness_bits, degree, relation_bits,
       ell_hat_bits, commitment_hash, iv, consistency_c,
       u_tilde, v_tilde[0..λ], correction)
squeeze() → relation_seed

第 3 轮打开挑战 opening_challenge：源自 relation_seed + 证明系数

absorb(relation_seed, proof.blinded_coeffs[0..d])
squeeze() → opening_challenge

然后通过 Grinding（递增 counter 使挑战值的后 w_grind 位为零）增加协议的安全性。

4.6 Grinding 证明工作量

#![allow(unused)]
fn main() {
fn challenge_has_required_grind_bits(challenge, w_grind) -> bool {
    // 检查挑战值的后 w_grind 个比特是否全部为零
    ((λ - w_grind)..λ).all(|bit| !bit_lsb(challenge, bit))
}
}

这要求 Prover 在最终确定签名前，平均需要进行 $2^{w_\text{grind}}$ 次尝试。对于 $w_\text{grind} = 8$（ReSolveD+ 的默认参数），Prover 需尝试约 256 次。这保证了 Verifier 对证明的非平凡性（non-triviality）有 $\lambda + w_\text{grind}$ 比特的置信度。

模块四：正则编码到多项式约束的转化——ANF 约束编译与蝴蝶隐式求值

一、概述

本模块是系统中最具理论创新性的部分。它解决的核心问题是：

如何将正则编码函数（Regular Encoding）——一个看似非线性的、基于查表的操作——转化为 degree-$c$ 的布尔多项式（ANF），然后在 ZK 协议的“隐式空间“中高效地安全求值？

这分为两个阶段：

  1. constraint_gen.rs — 编译阶段：离线预计算正则编码的 ANF 系数表
  2. implicit_eval.rs — 求值阶段：在线安全计算 χ-加权隐式多项式，使用蝴蝶折叠算法（Butterfly Evaluation）将复杂度从 $\mathcal{O}(m \cdot 2^c)$ 降至 $\mathcal{O}(m \cdot c)$

二、正则编码的代数化——从独热向量到 ANF 多项式

2.1 问题重述

正则编码将一个 $c$ 比特的输入 $X = (X_0, \dots, X_{c-1}) \in \mathbb{F}_2^c$ 映射为 $2^c$ 比特的独热向量：

$$ \text{RE}(X) = (f_0(X), f_1(X), \dots, f_{2^c-1}(X)) \in \mathbb{F}_2^{2^c} $$

其中 $f_{\text{idx}}(X) = 1$ 当且仅当 $\text{idx} = \sum_{t=0}^{c-1} X_t \cdot 2^{c-1-t}$（大端序整数解释）。

2.2 封闭形式的布尔多项式

每个输出位 $f_{\text{idx}}(X)$ 可以写为一个标准的 ANF（代数正规型） 多项式。其构造基于拉格朗日插值思想：

对于索引 idx，其大端二进制表示为 $j_0 j_1 \dots j_{c-1}$。定义：

$$ f_{\text{idx}}(X_0, \dots, X_{c-1}) = \prod_{t=0}^{c-1} (j_t + X_t + 1) \pmod{2} $$

解释：

  • 当 $j_t = 1$（索引的该比特为 1）：因子为 $X_t$（要求 $X_t = 1$）
  • 当 $j_t = 0$（索引的该比特为 0）：因子为 $1 + X_t$（要求 $X_t = 0$）

展开后得到一个 degree-$c$ 的 ANF 多项式——即布尔变量的乘积之和（XOR of ANDs），每个单项的度数 $\le c$。

示例（$c=3$, idx=5, 即二进制 101）： $$ f_5(X_0, X_1, X_2) = X_0(1+X_1)X_2 = X_0X_2 + X_0X_1X_2 $$

这是一个度数为 3 的多项式，包含 2 个单项式。

2.3 编译方法：子集包含（Möbius 变换）

代码中的 compile_re_constraints 没有直接展开上述乘积形式，而是使用更优雅的子集包含条件：

#![allow(unused)]
fn main() {
for idx in 0..domain_size {
    let mut coeffs = bitvec![u8, Msb0; 0; domain_size];
    for u in 0..domain_size {
        if (idx & u) == idx {
            coeffs.set(u, true);
        }
    }
    table.push(coeffs);
}
}
数学概念代码中对应
输出索引 $\text{idx}$idx: usize（0 到 $2^c-1$）
单项式索引 $u$（输入变量的子集指示器）u: usize
系数 $a_{\text{idx}, u} \in \mathbb{F}_2$coeffs[u]
$f_{\text{idx}}(X) = \bigoplus_{u} a_{\text{idx}, u} \cdot X^u$table[idx] 的每一行
条件 $(idx \mathrel{&} u) == idx$子集包含关系

为什么子集包含条件成立？

这是 Mobius 变换在布尔立方体上的应用。对于布尔函数 $f: \mathbb{F}_2^c \to \mathbb{F}_2$，其 ANF 系数由下式给出：

$$ a_u = \bigoplus_{x: \text{supp}(x) \subseteq u} f(x) $$

对于独热函数 $f_{\text{idx}}$，仅在 $x = \text{idx}$ 时取 1。因此 $a_u = 1$ 当且仅当 $\text{supp}(\text{idx}) \subseteq u$，即 $\text{idx} \mathrel{&} u = \text{idx}$。

例如对于 $c=3$, $\text{idx}=5$（二进制 101）：

  • $u=5$（101）：101 & 101 == 101 ✅ → 包含单项式 $X_0X_2$
  • $u=7$（111）：101 & 111 == 101 ✅ → 包含单项式 $X_0X_1X_2$
  • $u=4$（100）：101 & 100 != 101 ❌ → 不包含

三、隐式求值引擎：implicit_eval.rs

3.1 为什么需要“隐式“求值？

完整的 ZK 协议需要 Prover 证明约束系统 ${g_i(Y) = 0}_{i=1}^t$ 成立。如果不做优化，Prover 需要对 $t$ 个独立的约束多项式逐个求值、插值——这是 $\mathcal{O}(t \cdot 2^c)$ 的复杂度，对于 $t \approx 1280$、$c \approx 8 \sim 16$ 是不可接受的。

核心洞察：在协议中，Verifier 实际上只验证一个 χ-加权的总多项式：

$$ \tilde{G}(Y) = \sum_{i=1}^t \chi_i \cdot \tilde{g}_i(Y) $$

利用这个交换律，可以将求值复杂度从 $\mathcal{O}(t \cdot 2^c)$ 降低到 $\mathcal{O}(m \cdot c)$。

步骤 1：χ’ 聚合（求和交换）

对于约束系统 $g_i(Y) = \sum_j b_{i,j} f_j(Y) + y_i$，交换求和顺序：

$$ \sum_{i=1}^t \chi_i \cdot g_i(Y) = \sum_{j=1}^m \underbrace{\left(\sum_{i=1}^t \chi_i \cdot b_{i,j}\right)}{\chi’j} \cdot f_j(Y) + \underbrace{\sum{i=1}^t \chi_i \cdot y_i}{\text{const_term}} $$

compute_chi_prime 的极简实现：

#![allow(unused)]
fn main() {
fn compute_chi_prime(challenges, b_cols) -> Vec<F> {
    for each column j:
        chi_prime[j] = sum_{i: b_cols[j][i]==1} challenges[i]
}
}

对于 $\mathbb{F}_{2^{128}}$ 域，求和即 XOR。利用 BitVec::iter_ones() 快速扫描列为 1 的行索引。

compute_chi_prime_streaming_f128 的超高优化实现：

对于流式矩阵（StreamingMatrixCols），关键在于预计算字节子集 XOR 表以避免逐位扫描：

#![allow(unused)]
fn main() {
fn build_byte_subset_xor_table(challenges, byte_len) -> Vec<[u128; 256]> {
    // 对每个字节位置 byte_idx：构建 256 条掩码对应的 128-bit XOR 结果
    for each byte_idx:
        for mask in 1..256:
            subset_table[mask] = subset_table[mask & (mask-1)] ^ challenge[bit]
}
}

这利用了 Gray 码 / DP 递推：subset_table[mask] 表示对挑战值集合 challenge[byte_idx*8 + trailing_zeroes(mask)] 的 XOR 累加。一旦预计算完成，任何列的求值退化为 $n/8$ 次查表 + XOR：

#![allow(unused)]
fn main() {
fn subset_xor_from_bytes(column_bytes, table, byte_len) -> u128 {
    // 对列字节的每个字节位置，直接查表累加
    for each byte in column_bytes:
        acc ^= table[byte_idx][byte as usize]
}
}

更进一步，代码对手工展开的 $160$ 字节场景（$n=1280$ 的典型字节长度）做了 SIMD 风格的手动循环展开（xor_32! 宏同时处理 32 个字节，每次迭代处理 5 块共 160 字节），将循环开销降至最低。

并行化：当列数 $\ge 8192$ 时，自动启用 rayon 并行，每 256 列为一个 chunk，每个 worker 拥有独立的 column_bytes 缓冲区。

步骤 2：仿射基变换

将 witness 的每个比特 $w_i$ 和 VOLE 相关性 $v_i$ 组合成仿射基：

$$ \text{affine}_i(p) = v_i + w_i \cdot p $$

其中 $p$ 是当前的插值点。关键性质：

  • 在 $p=0$ 时：$\text{affine}_i(0) = v_i$（不泄漏 witness）
  • 在 $p\neq 0$ 时：witness 通过乘以 $p$ 隐藏其中

这由 implicit_evaluate_polynomials 中的以下代码实现：

#![allow(unused)]
fn main() {
let affine_bases: Vec<F> = v.iter()
    .zip(w.iter().by_vals())
    .map(|(v_i, w_i)| *v_i + if w_i { point } else { F::ZERO })
    .collect();
}

步骤 3：蝴蝶折叠（Butterfly Evaluation）

最核心的算法创新。

问题：对于每个大小为 $2^c$ 的分块，我们需要计算：

$$ \tilde{F}(p) = \sum_{j=0}^{\text{block_count}-1} \sum_{\text{idx}=0}^{2^c-1} \chi’{j, \text{idx}} \cdot \tilde{f}{\text{idx}}(p) $$

其中 $\tilde{f}{\text{idx}}(p)$ 是正则编码多项式 $f{\text{idx}}$ 在仿射基 $\text{affine}_{j}$ 上的求值结果。直接展开需要 $\mathcal{O}(2^c)$ 次域乘法和加法。

蝴蝶算法将复杂度降至 $\mathcal{O}(c)$。它的灵感来自 FFT 的 Cooley-Tukey 蝶形结构：

Algorithm: evaluate_chunk_butterfly(chi_prime_chunk[0..2^c),
                                     x_vars[0..c), point)

输入：χ'_j 数组（2^c 个 F_{2^128} 元素）
      x_vars（c 个仿射基）
      point（插值点 p）
输出：求和结果（1 个 F_{2^128} 元素）

初始化 work[i] = chi_prime_chunk[i]   (i = 0..2^c)

// 第 0 层（对应最低有效位 LSB）
// x_vars[c-1] 是 x_0（因为 x_vars 是大端序）
x_0 ← x_vars[c-1]
for i in 0..2^{c-1}:
    work[i] = work[2i] · (p + x_0) ⊕ work[2i+1] · x_0

// 第 k 层（k = 1..c-1）
for k in 1..c:
    x_k ← x_vars[c-1-k]     // 反向索引：大端转小端
    for i in 0..2^{c-1-k}:
        work[i] = work[2i] · (p + x_k) ⊕ work[2i+1] · x_k

返回 work[0]     // 最后只剩一个元素

数学原理：

对于 $c=2$ 的可视化示例：

        层 0 (LSB: x_0)          层 1 (x_1)
                                      
χ'_0 ──┐                           ──┐
       × (p+x_0)                    × (p+x_1)
χ'_1 ──┘ ──┐                      ──┘
           × (p+x_1)  ── work[0] 
χ'_2 ──┐   × x_1
       × x_0  ── work[1]
χ'_3 ──┘

work[0] = (χ'_0(p+x_0) ⊕ χ'_1 x_0)(p+x_1) ⊕ (χ'_2(p+x_0) ⊕ χ'_3 x_0) x_1
         = χ'_0(p+x_0)(p+x_1) ⊕ χ'_1 x_0(p+x_1) ⊕ χ'_2(p+x_0) x_1 ⊕ χ'_3 x_0 x_1
         = Σ_{idx=0}^{3} χ'_{idx} · f_idx(x_0, x_1)

这精确地计算了 χ-加权的正则编码多项式在点 $(x_0, x_1)$ 的求值。

为什么按 LSB→MSB 顺序？ 代码的注释解释了原因：

Butterfly 从 LSB 到 MSB 处理（与 (2*i, 2*i+1) 配对一致），因此遍历 x_vars 时需要反向：x_vars[c-1-k] 即对应 LSB 优先顺序。

这确保了 monomial_from_table_index(table_idx, block_start, c) 中的大端序变量索引与蝴蝶配对的正确对齐。

栈优化：当 $2^c \le 256$ 时，使用栈分配的定长数组[F; 256] 代替堆分配的 Vec，避免热路径上的内存分配。

完整流水线集成

implicit_evaluate_polynomials 函数组合了上述三个步骤：

#![allow(unused)]
fn main() {
pub fn implicit_evaluate_polynomials<F>(
    chi_prime: &[F],    // 步骤 1 输出
    const_term: F,       // Σ χ_i · y_i
    v: &[F],             // VOLE V 组件
    w: &MainBitSlice,    // witness 比特
    c: usize,            // 块大小
    point: F,            // 插值点
) -> F {
    // 步骤 2: 构建仿射基
    let affine_bases = v.iter()
        .zip(w.iter().by_vals())
        .map(|(v_i, w_i)| v_i + if w_i { point } else { F::ZERO })
        .collect();
    
    // 步骤 3: 蝴蝶折叠 + 常数项补偿
    evaluate_chunk_sum_from_affine(chi_prime, &affine_bases, c, point)
        + const_term * point.pow(c)
}
}

模块五：$\Pi^t_{dD-Rep}$ 核心协议与 NIZK 变换

一、概述

$\Pi^t_{dD-Rep}$（degree-$d$ Representation）协议是整个 ZK 系统的灵魂。它解决的核心问题是：

给定一个由 $t$ 个 degree-$d$ 布尔多项式组成的约束系统 ${g_i(X) = 0}_{i=1}^t$，Prover 如何在不泄露 witness $w \in \mathbb{F}_2^k$ 的前提下，让 Verifier 确信 $w$ 满足所有约束？

本模块包含两个层面：

  1. protocol_dd_rep.rs — $\Pi^t_{dD-Rep}$ 协议的三步实现（承诺 → 挑战 → 响应/验证）
  2. transcript.rs — Fiat-Shamir 变换的 Random Oracle 实例化（SHA-3 Keccak-256）

二、$\Pi^t_{dD-Rep}$ 协议三步曲

2.1 协议概览

$\Pi^t_{dD-Rep}$ 建立在 VOLE 相关性之上。回顾 Prover 和 Verifier 通过延迟 VOLE 功能 $\mathcal{F}_{\text{sVOLE}}$ 建立的关系：

$$ \begin{aligned} \text{Prover: } &(u, \mathbf{v}) \in \mathbb{F}2^k \times \mathbb{F}{2^{128}}^k \ \text{Verifier: } &(\Delta, \mathbf{q}) \in \mathbb{F}{2^{128}} \times \mathbb{F}{2^{128}}^k \end{aligned} $$

满足 $\mathbf{v} = \mathbf{q} + \Delta \cdot u$。Prover 拥有 witness $w \in \mathbb{F}2^k$，公开约束系统为 ${g_i(X)}{i=1}^t$。

│  (w, u, v)     │                 │  (Δ, q)         │
└────────┬────────┘                 └────────┬────────┘

2.2 Step 1：Witness 承诺（commit）

代数动机：Prover 不能直接发送 $w$（会泄露 witness），也不能发送 $u$（Verifier 已通过 $\mathcal{F}_{\text{sVOLE}}$ 部分知道 $u$）。解决方案是发送校正子（correction）：

$$ \text{corr} = w \oplus u \in \mathbb{F}_2^k $$

#![allow(unused)]
fn main() {
pub fn commit(&self) -> CommitmentMessage {
    let correction_bytes = self.witness.as_raw_slice()
        .iter().copied()
        .zip(self.u.as_raw_slice().iter().copied())
        .take(byte_len)
        .map(|(witness, u)| witness ^ u)
        .collect();
    CommitmentMessage { correction }
}
}

校正子的信息论安全性：由于 $u$ 是 Prover 私有的秘密向量（Verifier 只知道 $q$ 和 $\Delta$，但 $u$ 对 Verifier 具有 $-\log_2(\text{Pr}[u=0]) = k$ 比特的熵），$w \oplus u$ 完美隐藏了 $w$。

Verifier 端的吸收：Verifier 利用校正子修正自己的 $q$：

$$ \tilde{q}_i = q_i + \text{corr}i \cdot \Delta \in \mathbb{F}{2^{128}} $$

由于 $\mathbf{v} = \mathbf{q} + \Delta \cdot u$，且 $\text{corr} = w \oplus u$，验证：

$$ \tilde{q}_i = q_i + (w_i \oplus u_i) \cdot \Delta $$

当 $w_i = u_i$ 时，$\text{corr}_i = 0$，$\tilde{q}_i = q_i = v_i$ 当 $w_i \neq u_i$ 时，$\text{corr}_i = 1$，$\tilde{q}_i = q_i + \Delta = v_i$

因此 $\tilde{q} = \mathbf{v}$ 当且仅当校正子正确。Verifier 在不知道 $u$ 和 $v$ 的情况下，获得了正确 witness 所对应的 VOLE 关系。

常数时间实现：corrected_q_from_bitvec 使用 conditional_select（来自 subtle crate）逐字节展开校正子位，避免秘密依赖的分支。

2.3 Step 2：挑战生成（ChallengeMessage）

在交互式版本中，Verifier 采样随机域元素：

2.4 Step 3：Prover 响应（respond / respond_resolved_plus_implicit）

Prover 面对的核心任务是：计算 $\tilde{G}(Y) = \sum_{i=1}^t \chi_i \cdot \tilde{g}_i(Y)$ 在 $c+1$ 个插值点上的值，然后插值恢复多项式系数，最后用辅助 VOLE 进行盲化。

代码提供三种响应模式：

模式 A：通用 ANF 求值（respond）

直接对约束系统 ${g_i}$ 的每个多项式在每个插值点逐项求值：

Algorithm: respond(system, challenge)
1. 选择 degree+1 个插值点 p_0, ..., p_d
2. 预计算 point_powers[p][k] = p^k
3. 预计算 affine_bases[p][i] = v[i] + w[i]·p
4. 并行：
   对每个 χ_i 和对应的多项式 g_i:
       对每个插值点 p:
           计算 g_i 在仿射基下的值 × point_powers[d - degree]
       → 得到 d+1 个求值结果
       拉格朗日插值 → d 个系数
       每个系数 × χ_i 累加
5. 加上辅助 VOLE 的线性因子乘积 → 盲化系数
6. 返回 ProofMessage { blinded_coeffs }

关键子函数 evaluate_ck_at_point_precomputed：

#![allow(unused)]
fn main() {
pub(crate) fn evaluate_ck_at_point_precomputed<F: FiniteRing>(
    poly: &[Monomial],
    affine_bases: &[F],
    point_powers: &[F],
    degree_bound: usize,
) -> F {
    poly.iter().fold(F::ZERO, |acc, monomial| {
        let degree = monomial.len();
        let value = evaluate_monomial_affine_precomputed(monomial, affine_bases);
        acc + value * point_powers[degree_bound - degree]
    })
}
}

这里的数学是：对于 ANF 单项式 $X_{i_1} X_{i_2} \cdots X_{i_d}$，其在仿射基下的求值为：

$$ \tilde{X}i = v_i + w_i \cdot p \quad\Rightarrow\quad \prod{j=1}^{\text{deg}} \tilde{X}_{i_j} $$

然后乘以 $p^{d - \text{deg}}$ 补偿度数，使所有单项式对齐到同一度数 $d$。

模式 B：隐式求值（respond_resolved_plus_implicit）

对于使用正则编码的系统（ReSolveD+、环签名等），利用前文第四章的隐式求值引擎：

Algorithm: respond_resolved_plus_implicit(challenge, chi_prime, const_term, c)
1. 选择 c+1 个插值点 p_0, ..., p_c
2. 对每个 p:
    填充仿射基 affine[i] = v[i] + w[i]·p
    调用 implicit_evaluate_polynomials_from_affine(chi_prime, const_term, affine, c, p)
    → 求值结果 val_p
3. 拉格朗日插值恢复 c 个系数
4. 加上辅助 VOLE 盲化
5. 返回 ProofMessage { blinded_coeffs }

这里使用了第四章的蝴蝶折叠算法，复杂度为 $\mathcal{O}(k \cdot c)$ 而非 $\mathcal{O}(t \cdot 2^c)$。

模式 C：自定义求值器（respond_advanced_implicit）

允许应用层提供自定义求值函数 evaluator: Fn(&[F], F) -> F，用于环签名中需要同时求值两个矩阵的复杂场景。

2.5 Verifier 验证（verify / verify_resolved_plus_implicit）

核心验证等式：

$$ \sum_{i=1}^t \chi_i \cdot g_i(\tilde{q}) + \prod_{h=1}^{d-1} q_{\text{aux}, h} \stackrel{?}{=} \sum_{j=0}^{d-1} \tilde{a}_j \cdot \Delta^j $$

其中 $\tilde{q} = \mathbf{v}$（校正后），$g_i$ 是约束多项式，$\tilde{a}_j$ 是盲化系数。

Verifier 验证代码：

#![allow(unused)]
fn main() {
pub fn verify_resolved_plus_implicit(..., chi_prime, const_term, c, ...) -> Result<bool, String> {
    let lhs = implicit_evaluate_polynomials_from_affine(
        chi_prime, const_term, corrected_q, c, self.delta,
    ) + self.packed_aux_q.iter().copied().product::<F>();

    let rhs = evaluate_univariate(&proof.blinded_coeffs, self.delta);
    Ok(lhs == rhs)
}
}

逐项解析：

项数学公式代码
$\sum_i \chi_i \cdot g_i(\tilde{q})$约束加权和implicit_evaluate_polynomials_from_affine(chi_prime, const_term, corrected_q, c, delta)
$\prod_h q_{\text{aux}, h}$辅助 VOLE 乘积self.packed_aux_q.iter().copied().product::<F>()
$\sum_j \tilde{a}_j \cdot \Delta^j$单变量多项式求值evaluate_univariate(&proof.blinded_coeffs, self.delta)

evaluate_univariate 使用 Horner 法则：

#![allow(unused)]
fn main() {
fn evaluate_univariate<F>(coeffs: &[F], point: F) -> F {
    coeffs.iter().rev().fold(F::ZERO, |acc, coeff| acc * point + coeff)
}
}

等价于 $\tilde{a}_0 + \tilde{a}_1 \cdot \Delta + \tilde{a}_2 \cdot \Delta^2 + \cdots$，使用 Horner 法需要 $d$ 次乘法和 $d$ 次加法。

为什么 $\prod_h q_{\text{aux}, h}$ 会出现？

因为有 $d-1$ 个辅助 VOLE 相关性用于盲化（masking）。Prover 在响应中加入了 $\prod_h (v_{\text{aux},h} + u_{\text{aux},h} \cdot Y)$，Verifier 相应地在验证时加入 $\prod_h q_{\text{aux},h}$。

完整的正确性推导：

Prover 计算盲化系数： $$ \tilde{a}j = a_j + \sum{h: \text{组合}} (\text{来自线性因子乘积的 } Y^j \text{ 系数}) $$

Verifier 验证： $$ \sum_i \chi_i g_i(\tilde{q}) + \prod_h q_{\text{aux},h} = \sum_j \tilde{a}_j \cdot \Delta^j $$

当 $\tilde{q} = \mathbf{v}$ 且 witness 正确时，$\sum_i \chi_i g_i(\tilde{q}) = \sum_j a_j \cdot \Delta^j$，而线性因子乘积的盲化在两侧恰好抵消。

双层展开设计：

  1. Keccak-256 从 absorb 数据中压缩出 256-bit 种子
  2. ChaCha20 PRG 将种子扩展为任意长度的挑战值序列

模块六：基础应用层——ReSolveD+ 与环签名

一、概述

本模块是将底层 $\Pi^t_{dD-Rep}$ 协议应用到具体密码学原语的第一层。包含两种签名方案：

  1. ReSolveD+：基于 RSD（Regular Syndrome Decoding） 困难问题的数字签名方案。公钥是一个随机矩阵 $B$ 和其对应的综合症 $y$；私钥是一个随机秘密 $x$，满足 $y = B \cdot \text{RE}(x)$。
  2. 环签名（Ring Signature）：签名者可以从一个公钥集合（环）中匿名地签署消息，使验证者确信签名者属于该集合，但无法确定具体是哪一个。这里利用 Merkle 树累加器 + AFS 双矩阵哈希构造环。

二、ReSolveD+ 签名

2.1 数学基础：RSD 问题

RSD（Regular Syndrome Decoding） 是编码理论中的一个 NP-hard 问题：给定矩阵 $B \in \mathbb{F}_2^{n \times m}$ 和综合症 $y \in \mathbb{F}_2^n$，寻找一个正则词 $x$（其汉明重量极低，结构上由 $c$ 比特块映射为独热向量）使得：

$$ y = B \cdot \text{RE}(x) \quad (\text{在 } \mathbb{F}_2 \text{ 上}) $$

其中 $\text{RE}$ 是正则编码函数（见模块二），它将 $k$ 比特的输入 $x$ 映射为 $m = \frac{k}{c} \cdot 2^c$ 比特的独热拼接。

2.2 密钥结构

#![allow(unused)]
fn main() {
pub struct PublicKey {
    pub b_cols: StreamingMatrixCols,   // B 矩阵（流式列生成）
    pub y: MainBitVec,                 // 综合症 y = B · RE(x)
    pub k: usize,                      // 秘密 x 的比特长度
    pub c: usize,                      // 正则编码块大小
}

pub struct SecretKey {
    pub x: MainBitVec,                 // 秘密 x（witness）
}
}

密钥生成：

  1. 随机选择种子 $\sigma$ → 生成 $B = \text{PRG}(\sigma) \in \mathbb{F}_2^{n \times m}$
  2. 随机选择 $x \in \mathbb{F}_2^k$
  3. 计算 $y = B \cdot \text{RE}(x)$
  4. 公钥 = $(B, y, k, c)$，私钥 = $x$

2.3 ZK 证明语句

证明的目标是向 Verifier 零知识地证明“我知道 $x \in \mathbb{F}_2^k$ 使得 $y = B \cdot \text{RE}(x)$“。

等价地，Prover 需要证明 $x$ 满足以下约束系统 $g_i(x) = 0$：

$$ g_i(x) = y_i + \sum_{j=1}^m b_{i,j} \cdot f_j(x) = 0, \quad \forall i \in {1, \dots, n} $$

其中 $f_j(x)$ 是 $x$ 的正则编码 $\text{RE}(x)$ 的第 $j$ 个输出位，可以表示为 degree-$c$ 的 ANF 多项式（参见模块四）。

2.4 约束构建：build_resolved_plus_constraints

对于每个约束方程 $g_i(x) = 0$，代码使用三步流程构建 ANF 约束系统：

第一步：获取每个正则编码索引的本地单项式（local_monomials）

对于正则编码输出索引 $\text{idx}$（0 到 $2^c-1$），其 ANF 展开包含一组单项式（来自 re_table[idx] 中为 1 的列）。

第二步：反向映射 + BitVec 批量 XOR

#![allow(unused)]
fn main() {
// table_idx -> 包含它的 re_idx 列表
let mut table_idx_to_re_idx = vec![Vec::new(); domain_size];
for re_idx in 0..domain_size {
    for &table_idx in &local_monomials[re_idx] {
        table_idx_to_re_idx[table_idx].push(re_idx);
    }
}

// 对每个 (block_idx, table_idx)，计算 B·RE 中该单项式的系数行
for block_idx in 0..block_count {
    let block_offset = block_idx * domain_size;
    for table_idx in 0..domain_size {
        let mut presence = empty_col.clone();
        for &re_idx in &table_idx_to_re_idx[table_idx] {
            presence ^= &b_cols[block_offset + re_idx];  // BitVec XOR
        }
        global_monomial_presence.push(presence);
    }

三、环签名（Ring Signature）

3.1 数学基础：Merkle 累加器的 ZK 证明

环签名的核心思路是使用 Merkle 哈希树 累加环中所有公钥。签名者需要在零知识下证明自己知道一条从叶子（自己的密钥材料）到根（公开的 Merkle 根 $u_0$）的认证路径。

AFS 双矩阵哈希 用于 Merkle 树的每一层。对于层 $\theta$，给定左子节点 $v_\theta$ 和右子节点 $w_\theta$，双亲节点由下式算出：

$$ v_{\theta-1} = h_B(v_\theta, w_\theta) = B_0 \cdot \text{RE}(v_\theta) \oplus B_1 \cdot \text{RE}(w_\theta) $$

其中 $B_0, B_1 \in \mathbb{F}_2^{n \times m}$ 是两个独立的伪随机矩阵。

关键点：在每一层，签名者还需要证明路径的方向——即当前节点是左子还是右子。这通过一个方向比特 $j_\theta \in {0,1}$ 编码在约束中。

        + [v_2] + [w_2]           // 第 2 层的左右兄弟节点 (2n 比特)

其中 $j_\theta$ 表示从第 $\theta-1$ 层到第 $\theta$ 层的路径方向。例如 $j_1 = 0$ 表示 $u_0 = h_B(v_1, w_1)$（方向正常），$j_1 = 1$ 表示 $u_0 = h_B(w_1, v_1)$（左右交换）。

3.4 约束构建：build_ring_signature_constraints

约束系统包含 $(\ell + 1) \cdot n$ 个多项式（$\ell$ 层每层 $n$ 个 + 叶子层 $n$ 个）。

3.4.1 每层的约束方程

对于第 $\theta$ 层（$0 \le \theta < \ell$），我们知道：

$$ u_{\theta-1} = \begin{cases} h_B(v_\theta, w_\theta) & \text{若 } j_\theta = 0 \ h_B(w_\theta, v_\theta) & \text{若 } j_\theta = 1 \end{cases} $$

用 ANF 表示，这等价于：

$$ u_{\theta-1} = h_B(v_\theta, w_\theta) \cdot (1 + j_\theta) ; \oplus ; h_B(w_\theta, v_\theta) \cdot j_\theta $$

代码中，build_pair_hash_terms 为两种顺序分别计算 ANF 单项式集合 $A_i$（当 $j=0$）和 $B_i$（当 $j=1$）。然后通过 symmetric_difference 提取差异，用方向比特 $j_\theta$ “提升“差异单项式的度数：

#![allow(unused)]
fn main() {
let a_terms = build_pair_hash_terms(b0_cols, b1_cols, v_offset, w_offset, ...);
let b_terms = build_pair_hash_terms(b0_cols, b1_cols, w_offset, v_offset, ...);

for row_idx in 0..n {
    let mut equation = a_terms[row_idx].clone();
    // 对称差中的单项式需要乘以方向变量 j_theta（提升一度）
    for monomial in a_terms[row_idx].symmetric_difference(&b_terms[row_idx]) {
        let mut lifted = monomial.clone();
        lifted.push(direction_idx);   // 加入 j_theta 变量
        xor_insert(&mut equation, normalize_monomial(lifted));
    }
    // 处理常数项或父节点引用
    if theta_idx == 0 {
        // 顶层：目标根是公开的 root_u
        if root_u[row_idx] { xor_insert(&mut equation, Vec::new()); }
    } else {
        // 非顶层：目标父节点是上一层的 v 值
        xor_insert(&mut equation, vec![v_offset(theta_idx-1) + row_idx]);
    }
}
}

这精确实现了上述 $u_{\theta-1}$ 的 ANF 约束。

3.4.2 叶子层约束

叶子层方程为：

$$ v_{\ell-1} = h_B(x_0, x_1) = B_0 \cdot \text{RE}(x_0) \oplus B_1 \cdot \text{RE}(x_1) $$

fn main() {

}

Input:  affine (仿射基), point (p)

1. total = 0

     right_affine = affine[w_offset(theta) .. w_offset(theta)+n]

模块七：高级应用层——群签名与全动态属性基签名 (FDABS)

一、概述

本模块构建在环签名（模块六）的基础上，增加了两个重要的密码学功能：

  1. 群签名（Group Signature, GS）：除了环签名的匿名性外，群签名允许一个群管理员在必要时撤销匿名性，追踪出签名者的身份。这通过 McEliece 加密将身份信息 $j$ 和消息 $m$ 嵌入密文 $(c_1, c_2)$ 实现。

  2. 全动态属性基签名（Fully Dynamic Attribute-Based Signature, FDABS）：签名者拥有一个属性向量，只有当属性满足某个布尔电路策略（Policy） 时才能生成有效签名。代码将布尔电路编码为 AND 门约束，并利用 Merkle 树累加器证明属性集的有效性。

二、群签名（Group Signature）

2.1 安全问题域

群签名同时证明三个断言：

断言安全目标实现机制

        + [r_2]                                // McEliece 随机数 2 (ke-ell 比特)

        + [e'_2]                               // 噪声种子 2 (k 比特)

2.3 加密约束的构建

群签名约束系统 = 环签名约束 + 两个 McEliece 加密约束。

#![allow(unused)]
fn main() {
// 1. 复用环签名的 Merkle 路径约束

    b0_cols, b1_cols, root_u, ell, n, c_hash, hash_re_table

    append_linear_matrix_row_terms(
        &mut equation, g1_cols, r_len..r_len+ell, offsets.j_start, row_idx);

    constraints.push(equation.into_iter().collect());

// 3. 相同的结构用于 c2
}

加密约束的 ANF 形式：

对于密文 $c_1$ 的第 $i$ 个比特，约束为：

$$ c_{1,i} + \sum_{j: G_{1,i,j}=1} (r_1 | \text{bin}(j))_j + \text{RE}(e’_1)_i = 0 $$

其中：

  • append_linear_matrix_row_terms 处理 线性部分 $G \cdot \binom{r}{\text{bin}(j)}$：遍历 $G$ 矩阵的第 $i$ 行、涉及的列范围，每一列为 1 的位置对应一个线性变量
  • append_re_terms 处理正则噪声 $\text{RE}(e’)$：$\text{RE}(e’)_i$ 在约束层面被展开为多个 degree-$c_e$ 的 ANF 单项式的 XOR 求和（通过 append_re_terms 注入每个单项式）
  • 最后翻转常数项使方程等于 0

这与公式 $c = G \cdot \binom{u}{m} \oplus \text{RE}(e’)$ 完美对应——只不过这个关系被编码在约束方程 $g_i(\cdot) = 0$ 中，而不是直接计算。

3.5 叶奇偶性约束

FDABS 要求叶哈希的汉明重量为奇数。这是一个重要的合法性标记——它防止攻击者使用一个无效的（但 ANF 上满足的）叶节点替代真正的认证叶节点。

实现为单个方程：

#![allow(unused)]
fn main() {
let mut leaf_parity = BTreeSet::new();
for row_idx in 0..n {
    xor_insert(&mut leaf_parity, vec![leaf_v_offset + row_idx]);
}
xor_insert(&mut leaf_parity, Vec::new());  // + 1（常数项）
constraints.push(leaf_parity.into_iter().collect());
}

数学上表示：

$$ \left(\bigoplus_{i=0}^{n-1} v_{\ell,i}\right) + 1 = 0 \quad\Longleftrightarrow\quad \text{wt}(v_\ell) \equiv 1 \pmod{2} $$

测试 test_fdabs_constraints_reject_even_leaf_weight 验证了当叶哈希的汉明重量为偶数时约束被违反。

3.6 NAND 门二次约束（名为 AndGate，实际上实现 NAND）

代码中的门使用结构体名 AndGate，但约束实际实现的是 NAND（与非）门。对于每个门，要求 $x_{\text{out}} = \neg(x_{\text{in1}} \land x_{\text{in2}})$，在 $\mathbb{F}_2$ 上的 ANF 约束为：

$$ x_{\text{out}} + x_{\text{in1}} \cdot x_{\text{in2}} + 1 = 0 $$

验证：

  • $x_{\text{in1}} = 1, x_{\text{in2}} = 1$ 时：$x_{\text{out}} = 0$ ✓ (NAND)
  • $x_{\text{in1}} = 1, x_{\text{in2}} = 0$ 时：$x_{\text{out}} = 1$ ✓ (NAND)
  • $x_{\text{in1}} = 0, x_{\text{in2}} = 1$ 时：$x_{\text{out}} = 1$ ✓ (NAND)
  • $x_{\text{in1}} = 0, x_{\text{in2}} = 0$ 时：$x_{\text{out}} = 1$ ✓ (NAND)

（注意：NAND 门构成的电路与 AND 门一样具有表达能力——任何布尔电路都可以用 NAND 门单独构建。AndGate 结构体名反映了论文中的记法，但约束的实际代数实现是 NAND。）

#![allow(unused)]
fn main() {
// fdabs.rs 中 AND 门约束
for gate in &circuit.and_gates {
    let left_idx = wire_to_witness_index(offsets, circuit, gate.left_wire);
    let right_idx = wire_to_witness_index(offsets, circuit, gate.right_wire);
    let output_idx = wire_to_witness_index(offsets, circuit, gate.output_wire);
    constraints.push(vec![
        normalize_monomial(vec![left_idx, right_idx]),  // x_in1 · x_in2
        vec![output_idx],                                // + x_out
        Vec::new(),                                      // + 1 (常数)
    ]);
}
}

这对应于多项式的三个单项式：$X_{\text{in1}}X_{\text{in2}} + X_{\text{out}} + 1 = 0$。这是NAND 门（与非门）的 ANF 约束：当 $X_{\text{in1}} = 1, X_{\text{in2}} = 1$ 时，方程要求 $X_{\text{out}} = 0$；其他情况要求 $X_{\text{out}} = 1$。

3.7 最终输出约束

电路的最后要求是输出线 $x_{\text{out}}$ 必须为 1（策略满足）：

#![allow(unused)]
fn main() {
let output_idx = wire_to_witness_index(offsets, circuit, circuit.output_wire);
constraints.push(vec![vec![output_idx], Vec::new()]);
}

即 $X_{\text{output}} + 1 = 0$，等价于 $X_{\text{output}} = 1$。

二、安全参数分析：security.rs

2.1 安全性理论基础

$\Pi^t_{dD-Rep}$ 协议的安全性误差（soundness error）的上界为：

$$ \varepsilon \le 2^{-\text{combination_bits}} + d \cdot 2^{-\Delta_\text{bits}} $$

其中：

  • $\text{combination_bits}$：挑战空间 $\chi_i$ 的比特数（即 $t = n$ 个挑战的域大小）
  • $\Delta_\text{bits}$：Verifier 的差值挑战 $\Delta$ 的比特数
  • $d$：多项式的度数

2.2 安全性估计函数

#![allow(unused)]
fn main() {
pub fn polynomial_soundness_bound(
    degree: usize,
    combination_bits: usize,
    delta_bits: usize,
) -> Result<SoundnessEstimate, String> {
    let first_log2 = -(combination_bits as f64);               // log2(2^{-C})
    let second_log2 = (degree as f64).log2() - delta_bits as f64; // log2(d · 2^{-Δ})
    let max_log2 = first_log2.max(second_log2);
    let error_log2 = max_log2 + log2(2^{first-max} + 2^{second-max});
    Ok(SoundnessEstimate {
        degree, combination_bits, delta_bits,
        bit_security: -error_log2,
    })
}
}

使用数值稳定的 LogSumExp 技巧避免直接计算 $2^{-128} + d \cdot 2^{-128}$ 时的浮点下溢。

2.3 关键安全参数

polynomial_soundness_bound 确认了严格 128-bit 安全性无法达到——即使在理想情况下（两个域都是 128 比特），Forster 的上界也给出了约 124.68 比特的安全强度：

degree=9, combination=128, delta=128 → bit_security ≈ 124.678
degree=9, combination=128, delta=120 → bit_security ≈ 116.829

这与论文表 6 的 116-bit 保守估计一致。

Grinding 补偿：$w_\text{grind} = 8$ 比特的 grinding 提供了额外的 $\log_2(2^8) = 8$ 比特工作因子：

    self.bit_security + grinding_bits as f64
}
}

这使得实际攻击难度达到约 $116.8 + 8 \approx 124.8$ 比特。

4.3 par_chunks_mut 的列并行

在 χ’ 计算的热路径上，使用 par_chunks_mut 将列划分为连续 chunk，每个 worker 拥有独立的 column_bytes 缓冲区：

#![allow(unused)]
fn main() {
fn compute_chi_prime_streaming_f128_range_with_table(..., col_start, col_count) {
    if col_count >= CHI_PRIME_PAR_COL_THRESHOLD {
        chi_prime.par_chunks_mut(CHI_PRIME_PAR_COL_CHUNK)
            .enumerate()
            .for_each_init(
                || vec![0u8; byte_len],  // 每个 worker 独立缓冲区
                |column_bytes, (chunk_idx, chunk)| {
                    for (offset, chi) in chunk.iter_mut().enumerate() {
                        b_cols.fill_column_bytes_prechecked(base_col + offset, column_bytes);
                        *chi = subset_xor_from_bytes(column_bytes, table, byte_len);
                    }
                },
            );
    }
}
}

关键设计：for_each_init 为每个 Rayon worker 分配独立的列字节缓冲区，避免锁竞争和缓存行乒乓。
